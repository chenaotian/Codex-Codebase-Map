### 权限

#### 文件系统权限

thread/start 方法支持传入两种权限参数，分别为permissions参数和sandbox参数，这两种参数不可以同时传，他们是**新旧两个入口，归一到同一个权限模型**。这两个参数都支持三种权限策略，分别是：

- `read-only`：文件系统“只能读，不能写”；读范围默认是根级/全局可读范围。
- `workspace-write`：工作目录可写，但`.git`、`.agents`、`.codex` 等项目元数据默认会被保护成不可写/只读 carveout，避免模型修改 Codex 配置或仓库元数据。
  - workspace 是项目目录，cwd不是workspace，cwd只是用来解析相对路径。
- `danger-full-access`：**不使用 Codex 管理的文件系统沙箱**，而是拿到 **Codex 进程本身在操作系统里拥有的权限**，也就是权限跟codex 进程权限一样。

除此之外，permissions支持用户自定义的 `[permissions.<id>]`，用户可以在`$CODEX_HOME/config.toml`中配置，举例如：

```toml
default_permissions = "dev"

[permissions.dev]
description = "日常开发：可写工作区，禁读 env，禁网。"
extends = ":workspace"

[permissions.dev.filesystem]
glob_scan_max_depth = 2

[permissions.dev.filesystem.":workspace_roots"]
"." = "write"
"docs" = "read"
"**/*.env" = "deny"


```

含义是：

- `default_permissions = "dev"`：默认使用这个 profile。
- `[permissions.dev]`：定义 id 为 `dev` 的 profile。
- `extends = ":workspace"`：继承内置 workspace-write 基线。
- `"." = "write"`：允许写 workspace roots。
- `"docs" = "read"`：把 `docs` 限成只读。
- `"**/*.env" = "deny"`：禁止读取所有 env 文件。

如果你想额外加 workspace root，可以这样：

```toml
[permissions.dev.workspace_roots]
"D:/work/codex" = true
"D:/work/another-project" = true
```

#### 网络权限

除此之外，还有网络权限，网络权限和文件系统权限不同，代表这个会话/命令理论上能不能联网。像 `curl`、`npm install`、`git clone` 这类真实联网动作，默认会被约束或需要审批/提权。

网络配置例如：

```toml
禁用网络
[permissions.dev.network]
enabled = false

或部分允许
[permissions.review.network.domains]
"github.com" = "allow"
"api.github.com" = "allow"
```

### 沙箱

Codex 里的沙箱是对一个子进程加权限边界，它还是在本机跑，但读写、联网、进程能力被限制。

沙箱在不同平台实现不一样：

| 平台    | Codex 沙箱大致怎么做                                        |
| ------- | ----------------------------------------------------------- |
| macOS   | 用 `sandbox-exec` / seatbelt policy 限制文件和网络          |
| Linux   | 用 bubblewrap 构造文件系统视图，再配 seccomp / no_new_privs |
| Windows | 用 restricted token、ACL、WFP 网络过滤等方式约束进程        |
| 网络    | 可能通过 network-proxy、allowlist/denylist 管控             |
| patch   | 不一定启动进程，而是先做路径权限检查                        |

### 提权

Codex 的提权机制本质是“经审批后，允许某个命令绕过 Codex 沙箱或扩大沙箱权限”；审批决定能不能尝试，sandbox manager 决定是否真的能脱沙箱，OS 决定最终还能访问什么。

#### 基本权限

执行命令的时候的沙箱权限大概有四种，后三种涉及提权：

- `use_default`：按当前 thread 的权限画像和沙箱策略执行。
- `require_escalated`：请求这个命令 **脱离 Codex 沙箱** 执行。但不是获得root，而是 **Codex 当前进程所在 OS 用户的权限**。
- `with_additional_permissions`：请求的目标命令仍在沙箱里执行，但临时扩大文件/网络权限。
- `request_permissions`：审批对象不是针对单独命令，而是直接请求一组权限，批准后记录到 turn 或 session，后续命令仍在沙箱内执行，但沙箱策略会合并这些 granted permissions。

#### 提权发起

模型在发起命令类 tool call 时，可以携带，如：

```json
{
  "command": "git pull",
  "sandbox_permissions": "require_escalated",
  "justification": "需要网络访问远端仓库"
}
```

非 on-request 下，模型不能通过 tool 参数显式发起 `require_escalated`；但某些策略下，Codex 仍可能在沙箱失败后自动发起重试审批。主要区别就是on-failure，on-failure场景下模型不能一开始直接请求审批，而是要先在沙箱运行一次失败再尝试审批。

#### deny-read限制

Codex 明确认为：**denied reads only exist inside the sandbox**。如果当前权限画像里有 `deny` 路径，脱沙箱会让 deny 失效，所以 Codex 不允许直接无沙箱执行，如：

```toml
[permissions.dev.filesystem.":workspace_roots"]
"." = "write"
"**/*.env" = "deny"
```

这个情况下，即使模型请求提权，依然会在沙箱内执行，不能靠普通审批通过就直接读 `.env`。也就是说deny 是绝对无法绕过的禁止。
