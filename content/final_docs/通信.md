## 通信

![ChatGPT Image 2026年6月23日 16_48_10](D:\work\codex\final_docs\通信.assets\ChatGPT Image 2026年6月23日 16_48_10.png)

这里涉及到thread 内外的通信，分别有两个，外部事件进入thread走tx_sub，thread内部事件通知外部走rx_event

### tx_sub & rx_sub

`tx_sub` 和 `rx_sub` 是一对 `async_channel` 的两端，用来把外部操作投递进这个 `Codex` 会话。

```
tx_sub：输入入口。谁想让这个 Codex 会话做事，就往 tx_sub 发 Submission。
  -> 放进 Codex 结构体
  -> 给外部调用 Codex::submit(...)

rx_sub：执行入口。submission_loop 从 rx_sub 取 Submission，并串行分发给 Session 逻辑。
  -> move 到 tokio::spawn 的 submission_loop(...)
  -> 后台不断 recv 操作并执行
  
外部 / app-server / TUI / SDK
  -> CodexThread::submit(Op)
  -> Codex::submit(Op)
  -> Codex::submit_with_id(Submission)
  -> tx_sub.send(Submission)

后台 session_loop
  -> rx_sub.recv().await
  -> match Submission.op
  -> 调用对应 handler
  -> 修改 Session / 启动 turn / 中断 / 审批 / compact / shutdown
```

里面传输的内容就是上文提到的Submission，类型就是Op里面的类型。

### tx_event & rx_event 

`tx_event / rx_event` 是 **Codex 的输出事件通道**。把thread 内部产生的事件传递到外部，跟`tx_sub / rx_sub` 正好相反。

```
tx_event
  -> 传给 Session::new(...)
  -> 存到 Session 里
  -> Session 用它发送 Event

rx_event
  -> 放进 Codex 结构体
  -> 外部通过 Codex::next_event() 读取
  
Session / run_turn / tool call
  -> send_event(...)
  -> tx_event.send(Event)

Codex
  -> rx_event.recv()
  -> Codex::next_event()

CodexThread
  -> next_event()

app-server / TUI / SDK
  -> 读取 Event
  -> 转成 UI 消息 / JSON-RPC notification / terminal 输出
```

一般这些情况会产生event：

```
需要用户批准执行命令
需要用户批准 patch
需要用户回答问题
需要权限确认
工具开始/完成
模型输出增量
turn 开始/完成
报错/警告
token 统计
```
