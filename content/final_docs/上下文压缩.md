### run_pre_sampling_compact 

run_pre_sampling_compact 是再turn 开始前预测上下文长度已经接近上限后启动的上下文压缩入口。

- run_pre_sampling_compact 采样前是否需要压缩”的判断入口
  - 先调用 maybe_run_previous_model_inline_compact。如果上一轮用的是大上下文模型，这一轮切到小上下文模型，并且当前历史已经超过新模型可承受范围，它会先用“上一轮模型配置”触发一次压缩，原因标成 ModelDownshift，阶段标成 PreTurn。
  - 再调用 auto_compact_token_status 检查当前会话 token 是否达到自动压缩阈值。如果到了，就调用 run_auto_compact(..., DoNotInject, ContextLimit, PreTurn)。

### run_auto_compact 

run_auto_compact是上下文长度达到上限后自动触发的上下文压缩入口。

InitialContextInjection 参数：

InitialContextInjection ：压缩完成后，新的 replacement history 里要不要立刻重新插入 initial context，以及插在哪里。

- BeforeLastUserMessage：压缩后的 replacement history 里，要把 build_initial_context(...) 生成的 initial context 插到最后一个真实用户消息之前。
  - 使用场景，模型刚返回后，发现 token 达到压缩阈值，并且还需要继续下一次模型请求，所以必须把当前权限、环境、skills/plugins 等 initial context 立刻塞进 replacement history。
    - mid-turn auto compact，也就是run_turn中触发的压缩
- DoNotInject：压缩后的 history 里不主动插入 initial context。
  - 使用场景，这些场景压缩发生在一个正常 turn 开始前，或者独立 compact turn 里。压缩后会清掉 reference_context_item，下一次正常 run_turn 开头会发现 reference context 缺失，于是重新注入完整 initial context。
    - pre-turn auto compact，一轮开始的时候发现上下文上限了，在run_pre_sampling_compact 里
    - model downshift pre-turn compact，切换小上下文模型，也在run_pre_sampling_compact  里
    - manual /compact 手动压缩，这里手动压缩走run_compact_task

initial context 被插在最后一个真实 user message 或 summary 前面，而不是追加到末尾。

流程：

一共有remote_v2，remote，local三条路线可选，v2 和旧版remote 只是协议不同，都是请求openai 的特殊接口，local是不支持remote 压缩的时候直接用模型对话的方式压缩。**remote compact 大概率是 OpenAI 为 Codex 场景专门设计/优化过的压缩路径**，通常应该比 local 路线更可靠。

- run_auto_compact 是“真的去执行一次自动压缩”的统一分发器。
  - provider 支持 remote compaction 且开启 RemoteCompactionV2：走 remote_v2
    - pre-compact hooks hook点
    - clone 当前 history，如果 history 太大，先裁掉末尾 Codex 生成的 function/tool call 历史，避免 compact prompt塞不进上下文，如果真的塞不进去，会删掉一些codex 生成的辅助项，比如什么functioncall，toolsearch等的内容（没有东西可删会报错）。
    - 构造 prompt：当前 history +  tools(当前可用工具的模型可见声明) + base instructions(系统/开发者级基础指令)，这一步是为了让 compact 服务理解“当前线程是什么、之前发生过什么、有哪些工具语境、基础规则是什么”，Codex 把这些结构化信息放进 Prompt 里，然后发起 compact 请求。
    - 发起请求并从流里收集 **刚好一个** ResponseItem::Compaction；没有或多个都会报错
    - 构造 replacement history，过滤掉不该保留的 developer、伪 user、tool call 等
    - 按 initial_context_injection 决定是否重插初始 context
    - 替换 session history，重算 token
  - provider 支持 remote compaction 但没开 v2：走旧版 remote
    - 跟上面v2几乎一样，就请求方式等协议差别。
  - 否则走本地 inline compact
    - pre-compact hooks hook点
    - 增加本地inline compact提示词，然后发给模型让模型压缩
      - 如果 compact 请求本身超过 context window，会从 history 开头移除最旧 item 后重试
    - 从模型最后的 assistant message 取摘要，用 SUMMARY_PREFIX + 摘要 构造压缩 summary
    - 再在末尾拼接最近的用户完整消息，最多保留2000token，这里只保留用户消息，不保留模型输出。
    - 按 initial_context_injection 决定是否重插初始 context
    - 替换 session history，重算 token

### auto_compact_token_status & get_estimated_token_count 

- auto_compact_token_status 计算“当前会话是否已经达到自动 compact 阈值”

- get_estimated_token_count 本地从历史重新粗估一遍的诊断值。

### run_compact_task

手动压缩 /compact  调用的函数，最后走run_compact_task_inner，跟run_auto_compact 中的local路线一样。

```
run_auto_compact
  -> run_inline_auto_compact_task
    -> run_compact_task_inner(trigger=Auto, ...)

run_compact_task
    -> run_compact_task_inner(trigger=Manual, ...)
```

### 压缩区别举例

压缩前：

```
[
  developer: "你是 Codex。当前 cwd=... sandbox=...",
  user: "帮我分析 run_turn 主循环",
  assistant: "我先搜索源码...",
  tool_call: "rg run_turn",
  tool_output: "turn.rs:133 pub(crate) async fn run_turn...",
  assistant: "run_turn 的结构是：先 compact，再记录输入...",
  user: "继续分析 compact 逻辑",
  assistant: "compact 分为 pre-turn 和 mid-turn...",
  tool_call: "读取 compact.rs",
  tool_output: "很多源码内容...",
  assistant: "结论：run_auto_compact 是分发器..."
]
```

remote：

```
[
  user: "帮我分析 run_turn 主循环",
  user: "继续分析 compact 逻辑",
  Compaction {
    encrypted_content: "服务端生成的压缩状态/摘要内容"
  }
]
如果插入初始提示词：
[
  user: "帮我分析 run_turn 主循环",
  developer/context: "当前 cwd、sandbox、工具、模型配置...",
  user: "继续分析 compact 逻辑",
  Compaction { encrypted_content: "..." }
]
```

local：

```
[
  user: "帮我分析 run_turn 主循环",
  user: "继续分析 compact 逻辑",
  user: "<SUMMARY_PREFIX>\n目前已分析：run_turn 开始会调用 run_pre_sampling_compact；
        run_auto_compact 根据 provider/feature 分发到 remote_v2、remote 或 local；
        local compact 会用 compact prompt 让模型总结历史；
        已确认 turn.rs、compact.rs、compact_remote.rs 的关键调用点..."
]
```

 local compact 后很像“多了一条总结型用户消息”；remote_v2 后更像“多了一个专用的 compaction 结构体”。

除此之外，remote 压缩的上下文压缩后是**加密存储**的，所以我们不知道实际压缩的summary 具体内容。
