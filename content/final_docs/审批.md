## 审批

[toc]

![ChatGPT Image 2026年6月24日 15_29_43](D:\work\codex\final_docs\审批.assets\ChatGPT Image 2026年6月24日 15_29_43.png)

codex 的审批机制是一套“策略判断 + 沙箱选择 + 用户/自动审查 + 会话缓存 + 失败重试”的编排系统。审批系统围绕tool call设计：

```
tool call
  -> handler 规范化权限请求
  -> exec policy / patch safety 判断是否需要审批
  -> ToolOrchestrator 统一编排
  -> PermissionRequest hook
  -> Guardian auto-review 或用户审批
  -> 执行沙箱内尝试
  -> 必要时二次审批后无沙箱重试
  -> 缓存/持久化本次批准
```

### 审批策略

| policy                    | 含义                                                         |
| ------------------------- | ------------------------------------------------------------ |
| untrusted / UnlessTrusted | 不信任项目模式。只有“已知安全、只读”的命令可能自动放行，其他基本都要审批。通常用于 untrusted workspace。 |
| on-failure                | 老策略，已标注 deprecated。先让命令在 sandbox 里跑；如果失败，再请求批准无 sandbox / 更高权限重试。 |
| on-request                | 默认策略，模型/工具显式请求审批时才问。比如工具参数里带 require_escalated、with_additional_permissions、或某些操作按策略需要 approval。 |
| granular                  | 细粒度审批策略。不是一个简单模式，而是里面有多个开关，分别控制哪些审批流允许出现。比如沙箱/提权/mcp等。 |
| never                     | 永不请求用户审批。需要审批/提升权限的地方不会弹审批，能在沙箱就在沙箱，沙箱失败也不会弹提权审批。已经允许的操作继续允许。失败就直接返回给模型，也不会升级权限再试。 |

### 审批结果

- Approved：只批准这一次。
- ApprovedForSession：批准整个会话。
- ApprovedExecpolicyAmendment：批准并写入命令前缀规则。
- NetworkPolicyAmendment：批准或拒绝网络规则变更。
- Denied：拒绝。
- TimedOut：超时，按失败处理。
- Abort：中止当前任务。

审批场景：

- shell_command/exec_command：shell / exec 命令需要审批

- apply_patch：
  - apply_patch 先做 patch safety 判断，见 safety.rs (line 33)。如果修改路径不安全、沙箱不足、策略要求询问，就发apply_patch审批
  - 如果模型通过 shell 命令运行 apply_patch，会被 shell/unified exec handler 截获，改走 apply_patch 审批流，而不是当普通命令执行。

- 网络内联审批：managed network 拦截到未允许 host 时，会走网络审批服务。

- MCP tool call 相关审批，使用安装等。

- request_permissions ：请求在沙箱内临时扩大某些权限，比如多一个可写目录、允许某些网络访问，粒度更细。
- require_escalated：无沙箱执行命令请求审批

- Unix execve 拦截审批：Unix 下底层 execve 之前还会再触发一次审批，避免脚本里有什么奇怪的操作。
- 子agent 的请求也会转发给父session。

### 审查agent

审查agent是一个 Guardian 子 agent / 子会话，但它不是模型调用 spawn_agent 创建的普通 multi-agent 子 agent，而是 Codex 内部为了安全审查自动创建的专用子会话。

```
父 Codex session 创建
  └─ 只创建 GuardianReviewSessionManager，里面还没有 Guardian agent

第一次需要 Guardian 审查
  └─ 创建一个 guardian Codex 子会话，也就是 trunk review session
  	 -> 用 Guardian 专用 policy 作为 base instructions
  	 -> 清掉父会话 developer_instructions
 	 -> 设置 read-only / approval never / 禁用 apps、plugins、MCP 等
 	 -> 第一次审查时，把父会话 history 整理成一段 transcript 发给 Guardian

后续同一个会话里再需要 Guardian 审查
  └─ 优先复用这个 guardian 子会话

父 session 关闭 / 配置失效 / 临时审查完成
  └─ Guardian 子会话关闭
```

使用场景：

会启动 Guardian 审查 agent 的核心条件是：

```
某个动作原本需要审批
+ 当前 turn 配置为 approvals_reviewer = auto_review// 这里是对话中配置谁来审批
+ approval_policy 是 OnRequest 或 Granular
```

原本给用户审批的内容会给审查agent来审批。

审查agent 的提示词：

```
Guardian 子会话自己的初始 history：通常是空的
Guardian 子会话的初始指令：专门的 Guardian policy prompt
父会话 history：每次审查时被整理成 transcript，作为本次 user input 发给 Guardian
```
