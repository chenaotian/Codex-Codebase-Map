## 省流关键结论

- 核心：先进的agent 实际的harness 工程**大道至简**，实际没有做特别多的工作，能力主要来自背后**大模型的强大。**

- 会话相关

  - rollout文件约等于一个会话的存档，里面可以看到会话的历史，工具调用等很多细节，灵活运用可以实现很多东西，比如代码覆盖率统计。

  - turn 运行期间插话会进入pending_input，会再下一次给模型发消息之前把你插的这句话插入到history中。

  - 根据turn 与thread 的原理来看，如果codex 因为某些原因任务中断了，比如你不小心关掉了，那么直接跟他说继续工作就行了，不需要把指令重新发一遍。

  - plan 模式跟普通对话没区别只是增加一个标志位，然后输入提示词有区别，然后模型会输出一段结构化计划内容，本地会识别和显示。

- 上下文压缩相关：

  - 如果是账号登录，**openai 提供了特殊的上下文压缩通道**，如果用自己的模型，那么只能用默认的通道，相当于更大模型说：帮我压缩一下上下文。

  - **返回的压缩有的上下文摘要是加密的，本地看不了**，下次请求也是直接带加密的内容，服务端自己会解密，除此之外，完整思考内容也是加密的。

- subagent 相关：

  - 只有用户明确要求 sub-agents、子agent 或 parallel agent work这种提示词时才由 当前**父 agent 的模型发起。**

  - 在codex 层面上subagent 和普通的会话是一种数据结构，所以本质没啥区别。

  - **所以subagent 这个概念本质其实是给模型用的**，而不是给用户用的，用户如果想对subagent 进行操作其实不是很方便，需要用提示词引导模型来操作，用户层面直接启动thread或者fork thread能达到类似或者更好的效果(尤其是编写自己的agent 的时候)。

  - subagent 启动的时候是可以fork 模型当前历史的(当然也是模型调用的时候决定)，可以引导模型启动subagent 的时候使用这个功能，在一些场景会提供遍历。

- 安全相关：

  - 安全机制大概分为权限，审批和沙箱，用户提前配好允许的权限范围，codex 在执行的过程中会根据选择的审批策略触发审批，审批通过后会在之前配置的权限的沙箱中执行，如果权限不够执行失败，可能会重新尝试提权审批。

  - 用户批准的`ApprovedExecpolicyAmendment`类审批结果会在全局rules 中写入这条命令的规则，后续遇到不会再谈审批。

  - 替我审批功能其实就是启动一个子agent 来帮你审批，所以是大模型决定的。

  - 配置未deny 的目录/文件，即便模型请求提权，你通过了，也无法访问。

  - 提权不是提权到root，而是提权到codex 进程权限，只是脱离沙箱。

  - 可以自己在rules文件中增加免审批命令。

- 通过app-server可以实现但codex 客户端无法使用的功能

  - **dynamicTools 可以暴露自定义的工具**，比mcp轻量很多。

  - 使用thread/resume 接口可以自己通过一段虚构的hisstory启动一个会话。或拷贝一个rollout 文件后魔改然后用来resume会话，做到自定义history的效果。

  - outputSchema，turn/start接口可以通过这个参数来用json指定模型输出格式。

- 记忆相关：

  - 长期记忆默认不启用

  - 记忆总结也准备了特殊接口，但并没有实际使用

  - 长期记忆材料提取是实时进行的，但保存到记忆的工作每6小时才会进行一次

  - 每条长期记忆都是跟历史会话关联的，理事会话出问题对应的记忆也会被删除。

  - 记忆的使用是先给模型看摘要，模型根据需求查详细的记忆条目，记忆条目被模型使用会反馈，会记录每条记忆使用事件和次数，淘汰不好用的记忆。

- skill & MCP 相关

  - **skill其实就是一套准备好的提示词，核心步骤其实就一个，就是把skill.md塞到上下文里**，如果skill 还有其他文件，也都是模型根据skill.md的指引去使用。

  - MCP在大模型看来跟tool没区别

  - skill 如果重复点名是可以被重复加载的，为了节省token可以避免重复点名某skill

- 灵活使用hook点：

  - 可以帮你拦截危险命令

  - 可以防止大模型偷懒，大模型认为自己结束工作的时候可以通过hook自动检查它的工作，这里可以结合thread实现监工大模型的效果。跟goal类似

  - 帮你审批等。

- goal工具相关

  - goal 是把目标存到本地，所以不会收到上下文压缩影响，但检测到目标没完成turn却空闲，会强制拉起turn。

  - 大模型自己决定是否完成任务或阻塞任务。

- 很多功能的设计依赖模型能力：

  - 执行了后台任务，需要模型自己记得还有后台任务，自己主动去继续交互/继续获取输出，否则如果忘记，那么后台任务就没意义了。


  - goal 虽然是本地限制必须完成目标，但目标完成的标准还是大模型自己来决定的，模型能力不行还是可能偷懒。


  - 一个turn 是否结束取决于模型是否输出结束信号，所以一个工作做到什么程度取决于模型能力，比如修复完bug是否需要测试，是否需要更新git等，几乎全取决于模型能力。
