### 记忆系统简介

| 分类     | Codex 里的对应物                                             | 特点                                                |
| -------- | ------------------------------------------------------------ | --------------------------------------------------- |
| 静态记忆 | 全局 `AGENTS.md`、项目/目录级 `AGENTS.md`、`AGENTS.override.md`、`config.toml`、固定项目文档 | 人写的、稳定的、声明式规则或偏好                    |
| 动态记忆 | 当前上下文窗口、当前 thread 历史、工具结果、环境信息、compaction summary | 随会话推进变化，主要服务当前任务                    |
| 长期记忆 | `~/.codex/memories/`、`memories_1.sqlite` 里的 `stage1_outputs`、`memory_summary.md`、`MEMORY.md`、`rollout_summaries/`、memory `skills/` | Codex 从历史线程中抽取/合并总结出来的，可跨线程复用 |

- `AGENTS.md` 算是静态记忆，但**不是模型自己学出来的，而是人(或人让模型)显式写下来的规则**

- Codex 的长期记忆 **默认关闭**。官方 Memories 页明确说 “Memories are off by default”，开启方式是在 Codex app 设置里打开，或在 ~/.codex/config.toml 的 [features] 里加：

  ```toml
  [features]
  js_repl = false
  memories = true
  ```

  如果你希望 Codex 在不同线程之间记住稳定偏好、常用工作流、技术栈、项目约定、已知坑点，减少每次重复交代上下文，就适合开启。**官方也强调：必须稳定执行的团队规则仍应放在 AGENTS.md 或仓库文档里，长期记忆只是“本地辅助回忆层”，不要当成唯一规则来源。**

### 记忆系统分类

| 类型     | 本质                                     | 谁写入               | 存在哪里                                         | 什么时候读                              | 主要用途                                |
| -------- | ---------------------------------------- | -------------------- | ------------------------------------------------ | --------------------------------------- | --------------------------------------- |
| 长期记忆 | 从历史对话中提炼出的持久化经验/偏好/事实 | 模型后台生成         | SQLite + `$CODEX_HOME/memories/`                 | 新 thread / 相关任务时读                | 跨会话记住长期有用的信息                |
| 静态记忆 | 用户手写的固定指令                       | 用户/项目维护者      | `AGENTS.md`、`AGENTS.override.md`                | session/run_turn 建初始上下文时自动注入 | 告诉 Codex 项目规则、代码风格、测试方式 |
| 动态记忆 | 当前 thread 的上下文状态                 | 对话运行过程自动累积 | conversation history / rollout / context manager | 每轮模型请求都会带相关上下文            | 让模型记住当前对话里刚发生的事          |

#### 长期记忆

长期记忆是最像“真正 memory system”的那套。它会从旧 thread 里抽取可能有长期价值的信息，先写入 `stage1_outputs`，再经过 Phase 2 汇总到 `$CODEX_HOME/memories/` 下的文件，比如 `MEMORY.md`、`memory_summary.md`、`rollout_summaries/*`。

#### 静态记忆

静态记忆主要就是 `AGENTS.md` 这一类。它不是模型总结出来的，而是用户或项目维护者手写(或让模型写)的规则文件。

#### 动态记忆

动态记忆就是实时上下文。

### 承载记忆的实体(文件 & 数据库)

**1. 专用记忆数据库**：$CODEX_HOME/memories_1.sqlite

- 这是长期记忆 pipeline 的核心 DB。

- 里面主要两张表：

  - stage1_outputs：Phase 1 从历史会话抽出的记忆内容
    字段包括 raw_memory、rollout_summary、rollout_slug、usage_count、last_usage、selected_for_phase2。

  - jobs：Phase 1 / Phase 2 的任务队列、锁、租约、重试状态。

**2. 主状态数据库**：$CODEX_HOME/state_5.sqlite

- 它主要是线程/会话索引和运行态元数据，但其中的threads表里存放了一些跟记忆管理相关的信息：

  - threads.memory_mode：enabled / disabled / 内部 polluted

  - threads.rollout_path：指向原始会话 JSONL

  - updated_at、source、cwd、git metadata 等，供 Phase 1 判断哪些 thread 可抽取

**3. 原始会话记录，也就是 Phase 1 的输入**：$CODEX_HOME/sessions/YYYY/MM/DD/rollout-...jsonl

- 这些 JSONL 只是作为类似会话日志的形式存在，但它们是记忆生成的原料。Phase 1 会通过 threads.rollout_path 找到这些 rollout 文件，然后从里面过滤消息、工具调用、输出等，抽取成 stage1_outputs。

**4. 文件化长期记忆目录**：$CODEX_HOME/memories/

- 这是模型后续真正读取的长期记忆工作区，主要文件/目录：

  - **memory_summary.md：最精简、会被直接注入 developer prompt 的摘要。**

  - MEMORY.md：可搜索的主索引/注册表。

  - raw_memories.md：由 DB 中选中的 stage1_outputs.raw_memory 合并生成。

  - rollout_summaries/*.md：每个被 Phase 2 选中的 rollout 一份摘要。

  - skills/<skill-name>/SKILL.md 等：长期沉淀出的技能化记忆。

  - extensions/：扩展记忆输入/派生内容区域。

  - extensions/ad_hoc/notes/：显式要求添加记忆时写入的临时/手工 note 区域。


**5. Phase 2 工作区状态文件**：$CODEX_HOME/memories/.git

- memories/ 自身被当成一个 git-baseline 工作区，用来判断本轮 Phase 2 同步后文件树发生了什么变化。

- 还有一个临时文件：$CODEX_HOME/memories/phase2_workspace_diff.md
  - 它是 Phase 2 consolidation agent 读取的 diff 输入，成功后会在 reset baseline 前移除。
